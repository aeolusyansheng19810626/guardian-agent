# 交付物检查标准

## 1. 文档结构完整性

系统需求/设计类文档建议必填章节：

- 文档目的与范围
- 业务背景 / 用户场景
- 功能需求
- 非功能需求（性能 / 可用性 / 安全 / 兼容性 等）
- 系统架构 / 模块划分
- 数据模型 / 接口定义
- 风险与依赖
- 验收标准 / 验证方式

缺少**风险章节**视为高严重度问题。

## 2. 需求描述可测试性

- 每条需求必须可以被验证、可以判定合格/不合格。
- **不接受**模糊描述：
  - "性能要好""响应要快""界面友好""稳定可靠"
  - "支持大并发""高可用"（无量化指标）
- **应改为**：
  - 性能 → "P95 响应时间 ≤ 500ms，QPS ≥ 1000"
  - 可用性 → "月度可用率 ≥ 99.9%"
  - 兼容 → "支持 Chrome ≥ 100、Edge ≥ 100"

## 3. 风险章节必要性

- 必须列出已识别风险、影响、概率、缓解策略、责任人。
- 仅写"暂无风险"视为不合格。

## 4. 接口与数据规范

- 接口需说明协议、入参、出参、错误码、鉴权方式。
- 数据模型需有字段类型、约束、必填性、字段说明。

## 5. 格式规范

- 标题层级清晰、不跳级（H1→H2→H3）。
- 表格用于结构化数据，不滥用代码块。
- 图示有图号与说明文字。
- 术语首次出现需定义；缩写有对照表。

## 6. 一致性与可追溯

- 需求 ID 与设计、测试可追溯。
- 同一概念前后术语统一。

## 7. 评分扣分参考

| 缺陷类型 | 严重度 |
|---|---|
| 缺风险章节 | high |
| 非功能需求未量化 | high |
| 功能需求不可测试 | high |
| 接口/数据模型缺关键要素 | med |
| 标题层级混乱 | low |
| 术语前后不一致 | med |
| 图示无说明 | low |

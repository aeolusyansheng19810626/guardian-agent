# 障害报告检查标准

## 1. 障害概要（必填要素）

- **发生时刻**：精确到分钟（YYYY-MM-DD HH:MM）。
- **检知时刻**：发生与检知时刻不一致时需分别记载。
- **恢复时刻**：完全恢复正常服务的时刻。
- **影响时长**：自动从上述三个时刻派生，但应明确写出。
- **影响范围**：受影响的系统、子系统、功能模块。
- **影响用户数 / 影响业务**：定量描述（人数、件数、金额）；不可仅写"少数用户"。

## 2. 现象与时序

- 客户/外部视角看到的具体现象（错误码、异常画面、超时等）。
- 时序日志：从异常征兆 → 检知 → 临时対応 → 恢复，按时间顺序列出关键事件。

## 3. 根本原因分析（最关键）

- 必须给出**具体技术原因**，例如："索引缺失导致全表扫描""配置文件中超时阈值设置过小"。
- **不接受**以下表面描述：
  - "服务器异常""系统故障""网络问题"
  - "操作失误""人为原因"（必须说明误操作的具体步骤与防呆缺失）
  - "性能不足"（必须给出指标与瓶颈位置）
- 推荐采用 5-Why 或 Fishbone 方法逐层追问。
- 区分**直接原因**（触发本次问题的事件）与**根本原因**（深层缺陷）。

## 4. 临时対応 vs 恒久対応

- **临时対応**：本次故障已采取的、用于快速恢复的措施（重启、回滚、切流等）。需注明执行人与时刻。
- **恒久対応**：消除根因、防止再发的根本措施。必须区别于临时対応。

## 5. 再発防止策（必含三要素）

每条防止策必须包含：

1. **具体措施**：可执行、可验证（例："在 CI 中加入索引覆盖检查脚本"，而非"加强代码评审"）。
2. **担当者**：具体到人或团队。
3. **期限**：明确的完成日期。

抽象口号（"加强监控""提升意识""完善流程"）若无以上三要素，视为不合格。

## 6. 影响波及与横展开

- 类似的代码、配置、运用是否在其他系统/模块存在；横展开调查范围与结果。

## 7. 评分扣分参考

| 缺陷类型 | 严重度 |
|---|---|
| 缺少根本原因或表面化描述 | high |
| 防止策缺三要素之一 | high |
| 临时対応与恒久対応混淆 | med |
| 缺影响用户数 / 时长 | med |
| 时序信息不完整 | low |
| 横展开调查缺失 | med |
